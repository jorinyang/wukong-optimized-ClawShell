## 第一章：系统概述与定位

### 1.1 ClawShell 核心定义

**ClawShell本质上是一个适用于类OpenClaw架构的增强型外骨骼功能插件；其中具备的自感知、自适应、自组织能力是以插件的形式加强类OpenClaw架构的底层能力，使其能够更快了解环境、适应环境、动态协作。**

#### 自感知

自感知是为了获取ClawShell所在的设备、局域网络、需要接入/已接入的有凭证权限的云端、公共互联网所包含的环境信息和主体的能力、权限边界信息。

| 感知维度 | 内容 |
|----------|------|
| **本地环境感知** | 设备性能指标、操作系统版本、进程运行状态、计算资源消耗 |
| **网络环境感知** | 局域网内服务发现、端口开放状态、网络拓扑结构、设备互联关系 |
| **云端凭证感知** | 已接入云服务的权限范围、API密钥有效性、数据访问边界、服务配额状态 |
| **互联网感知** | 外部服务可用性、域名解析状态、网络链路质量、公共API响应能力 |

#### 自适应

自适应的边界是ClawShell根据自身的环境依赖要求和数据规范要求对所感知到的环境进行改造，目的是实现在感知范围内掌控能获得的最大权限与能力；如果无法改变环境，或者改变环境无法达到自适应目的，为了适应所感知到的环境，根据已获知的信息进行自我改造。

| 自适应层次 | 说明 |
|------------|------|
| **环境改造层** | 当ClawShell检测到环境不满足其运行依赖时，主动调整环境配置使其符合要求 |
| **自我改造层** | 当环境无法被改变或改造无法达成自适应目的时，根据已获知的环境信息调整自身行为模式 |

#### 自组织

自组织是根据任务目标进行最佳实践的计划和执行能力；根据自感知到的信息，结合自适应掌控的权限与能力，匹配合适的任务执行方案，制订最优计划并执行，执行过程中动态调度所掌控的资源支持完成任务目标。

### 1.2 核心设计哲学

**ClawShell遵循原则：**

| 原则 | 说明 |
|------|------|
| **异构同效** | 不同架构、不同技术栈的模块，能在同一机制下发挥同等效能 |
| **异步优先** | 长时间任务异步执行，避免阻塞主流程 |
| **低耦合** | 模块间通过文件协议和EventBus通信，而非直接函数调用 |
| **高鲁棒** | 多层级错误恢复、守护进程保活、自动降级 |
| **高泛用** | 感知层抽象、适配器模式、标准化接口 |
| **高协同** | EventBus + ContextManager + TaskMarket + Swarm |
| **高扩展** | 新模块必须可感知、可适应、可协作 |

### 1.3 架构定位

**ClawShell定位为OpenClaw的外骨骼增强层，以无侵入方式叠加新能力，不修改OpenClaw核心代码，不依赖OpenClaw内部实现，通过标准化接口与OpenClaw交互。**

这种外骨骼定位确保了ClawShell与OpenClaw的版本解耦，OpenClaw的升级不会影响ClawShell的正常运行，ClawShell的升级也不会破坏OpenClaw的稳定性。两个系统各自独立演进，通过明确接口定义实现协同。

### 1.4 封装原则 (v0.9)

**ClawShell的封装设计遵循五大原则，确保跨环境可移植性和版本兼容性。**

| 原则 | 说明 |
|------|------|
| **无侵入** | 不修改任何已部署框架的核心代码，保护原始系统完整性 |
| **低耦合** | 模块间通过标准化文件协议和EventBus通信，避免直接依赖 |
| **可移植** | 支持macOS/Linux，Windows兼容可选，适应多平台环境 |
| **幂等性** | 重复安装不对已有配置产生副作用，保障系统安全 |
| **版本解耦** | ClawShell版本升级不影响已部署框架，独立演进互不影响 |

### 1.5 指导思想：工程控制论

**ClawShell的架构设计以工程控制论为指导思想，核心是信息反馈、动态调控和系统整体思维。**

| 控制论概念 | ClawShell实现 |
|------------|---------------|
| **系统整体思维** | 追求整体合理性而非局部最优 |
| **信息反馈机制** | EventBus + ContextManager + Comparator |
| **动态调控** | AdaptiveParameterTuner + RobustController |
| **鲁棒性设计** | PerturbationDetector + 备份恢复机制 |
| **层次化建模** | 四层架构 + Phase执行计划 |
| **综合集成** | HumanMachineValidation + ExpertKnowledgeInterface |

## 第八章：双脑协同架构

### 8.1 设计理念

**双脑协同模型**：OpenClaw作为"后脑"（执行引擎）+ Hermes作为"前脑"（进化引擎），通过EventBus实现双向通信。

### 8.2 架构分层

```
Layer 5: 协同调度层
┌────────────────────────────────────────────────────────────┐
│  HermesBridge (Phase 1 实现)                             │
│  ├─ PriorityClassifier  优先级分类                       │
│  ├─ ResponseModeMatcher 响应模式匹配                    │
│  ├─ EventBusSubscriber 订阅OpenClaw事件                │
│  ├─ EventBusPublisher  发布Hermes事件                   │
│  └─ MessageQueue       优先级消息队列                   │
└────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│  EventBus 双向通信                                       │
│  ├─ clawshell.task.*    OpenClaw → Hermes                │
│  ├─ clawshell.error.*  错误事件                         │
│  └─ hermes.insight.*   Hermes → OpenClaw                 │
└────────────────────────────────────────────────────────────┘
                           │
         ┌─────────────────┴─────────────────┐
         ▼                                   ▼
┌─────────────────────┐             ┌─────────────────────┐
│     OpenClaw        │             │       Hermes         │
│      (后脑)          │             │       (前脑)         │
│  ├─ TaskExecutor   │             │  ├─ ReviewEngine    │
│  ├─ EventPublisher│             │  ├─ InsightGen     │
│  └─ SkillLoader   │             │  ├─ SkillGenerator  │
└─────────────────────┘             └─────────────────────┘
```

### 9.2 架构

```
凌晨5:00 ─┬─→ self_repair_detector.py ─→ repair_queue.json
          │          ↓
          │    27项全景检测
          │          ↓
          └────────────────────→ 每10分钟执行1个阶段
                                        ↓
                               self_repair_executor.py
                                        ↓
                               修复动作库 (20+动作)
```
